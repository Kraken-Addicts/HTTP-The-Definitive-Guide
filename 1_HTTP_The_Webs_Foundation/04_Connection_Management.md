### 4.2 　 TCP의 성능에 대한 고려　 `yeosong`
- HTTP가 TCP 바로 윗 계층이므로, HTTP 트랜잭션의 성능을 논할 때 TCP 성능을 고려할 필요가 있다.

#### 4.2.1 HTTP 트랜잭션 지연

##### 트랜잭션이란?
- 하나의 논리적 기능을 수행하기 위한 작업의 단위
- 작업의 완전성을 보장함
  - 성공시 커밋
  - 실패시 롤백(방금 수행 실패한 트랜잭션을 이전 상태로 되돌리고, 재시도 혹은 폐기함.)
- HTTP transaction 1개는 1개의 HTTP request와 그에 상응하는 HTTP response로 이루어져있다.
  - [(이 페이지를 참고했어요)](https://www.informit.com/articles/article.aspx?p=32079)

----------------------
- HTTP 지연은 클라이언트나 서버가 너무 많은 데이터를 내려받거나 복잡하고 동적인 자원들을 실행할 때 생긴다. (O/X)
- HTTP 지연의 대부분은 TCP 네트워크 지연 때문에 생긴다. (O/X)
- TCP 네트워크의 지연 정도는 요청/응답 메시지의 크기, 클라이언트와 서버간의 거리에 따라 달라진다. (O/X)
- TCP 네트워크의 지연은 TCP 프로토콜의 기술적 복잡성의 영향을 받는다. (O/X)

#### 4.2.2 성능 관련 중요 요소

##### 4.2.3 TCP 커넥션 핸드셰이크 지연
- 새로운 TCP 커넥션을 열면 TCP 커넥션 
- 아주 큰 데이터를 주고받지 않는 평범한 HTTP 트랜잭션은 50% 이상의 시간을 TCP를 구성하는 데 쓴다.
- TCP 구성으로 인한 지연을 제거하기 위해 이미 존재하는 커넥션을 재활용하기도 한다.

### 4.6 　 파이프라인 커넥션　 `kukim`
1. http 클라이언트는 지속커넥션 여부와 상관없이 파이프라인을 연결할 수 있다.(O/X)
  - 정답 : X
    - 지속커넥션인지 확인하기 전까지는 파이프라인을 이어선 안된다.
2. http 클라이언트는 지속 커넥션 응답에 대해 요청 순서와 상관없이 Content-Length을 통해 정렬하여 해석할 수 있다. (O/X)
  - 정답 : X
    - http 응답은 요청 순서와 같게 와야한다. http 메세지는 순번이 매겨져 있지 않아서 응답이 순서없이 오면 순서에 맞게 정렬 시킬 방법이 없다.
3. http 지속 커넥션의 요청에 대한 서버의 응답 메세지가 어떠한 오류로 끊어지더라도 서버에서 이를 기억하여 다시 보내준다. (O/X) 
  - 정답 : X
    - http 클라이언트는 커넥션이 언제 끊어지더라도 완료되지 않은 요청이 파이프라인에 있으면 언제든 다시 요청을 보낼 준비가 되어야 한다.
4. 모든 메서드는  파이프라인 커넥션을 사용할 수 있다. (O/X)
  - 정답 : X
    - 멱등한 메서드만 사용할 수 있다. 비멱등(nonidempotent)한 메서드를 사용하여 에러가 발생하면 파이프라인을 통한 요청 중 어떤 것들이 서버에서 처리가 되었는지 알 수 없기 때문에 요청을 재차 보내면 문제가 생길 수 있다. 
5. (+a) 현재 HTTP/1.1 파이프라이닝은 모던 브라우저에서 기본적으로 활성화되어 사용되고 있다. (O/X)
- 정답 : O
  - HTTP 파이프라이닝은 모던 브라우저에서 기본적으로 활성화되어있지 않다.
  - 왜냐하면 정확히 구현해내기 어렵기 때문이다. 전송 중인 리소스의 크기, 효과적인 RTT(Round-trip delay, 왕복지연시간),  대역폭 크기의 설정, HOL(Head of Line Blocking, 헤드라인차단, 첫 번째 패킷이 응답이 올 때 까지, 이전 요청이 완료 될 때 까지 기다리는)으로 문제가 된다.
  - 파이프라이닝은 HTTP/2로 오면서 멀티플렉싱으로 대체 되어 사용된다.
  - 한 줄 요약 : ***파이프라인 커넥션은 잘 사용하지 않는다.*** ~~~(왜 공부한거니)~~~
