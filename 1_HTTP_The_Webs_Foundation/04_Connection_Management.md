### 4.2 　 TCP의 성능에 대한 고려　 `yeosong`
> HTTP가 TCP 바로 윗 계층이므로, HTTP 트랜잭션의 성능을 논할 때 TCP 성능을 고려할 필요가 있다.

#### 4.2.1 HTTP 트랜잭션 지연

##### 트랜잭션이란?
> - 하나의 논리적 기능을 수행하기 위한 작업의 단위
> - 작업의 완전성을 보장함
>   - 성공시 커밋
>   - 실패시 롤백(방금 수행 실패한 트랜잭션을 이전 상태로 되돌리고, 재시도 혹은 폐기함.)
> - HTTP transaction 1개는 1개의 HTTP request와 그에 상응하는 HTTP response로 이루어져있다.
>   - [(이것을 참고했어요)](https://www.informit.com/articles/article.aspx?p=32079)

- HTTP 지연은 클라이언트나 서버가 너무 많은 데이터를 내려받거나 복잡하고 동적인 자원들을 실행할 때 생긴다. (O/X)
- HTTP 지연의 대부분은 TCP 네트워크 지연 때문에 생긴다. (O/X)
- TCP 네트워크의 지연 정도는 요청/응답 메시지의 크기, 클라이언트와 서버간의 거리에 따라서도 달라진다. (O/X)

#### 4.2.2 성능 관련 중요 요소

##### 4.2.3 TCP 커넥션 핸드셰이크 지연
- 아주 큰 데이터를 주고받지 않는 평범한 HTTP 트랜잭션은 50% 이상의 시간을 TCP를 구성하는 데 쓴다. (O/X)
- TCP 구성으로 인한 지연을 제거하기 위해 이미 존재하는 커넥션을 재활용하기도 한다. (O/X)

##### 4.2.4 확인응답 지연
- TCP는 성공적인 데이터 전송을 보장하기 위해 확인응답 패킷을 사용한다. (O/X)
- '확인응답 지연' 알고리즘은 확인응답 패킷을 잠시 버퍼에 둔 채로 편승(piggyback)할 수 있는 데이터 패킷를 찾는 알고리즘이다. (O/X)

##### 4.2.5 TCP 느린 시작(slow start)
- 인터넷의 급작스러운 부하와 혼잡을 방지하기 위하여 커넥션의 최대 속도를 제한하여 점차 높여 나가는 것을 'silly window syndrome'이라고 부른다. (O/X)
- 패킷이 성공적으로 전달되는 각 시점에 송신자는 추가로 4개의 패킷을 더 전송할 수 있는 권한을 얻는다. (O/X)
- 새로운 커넥션은 항상 튜닝된 커넥션보다 빠르다 (O/X)

##### 4.2.6 네이글(Nagle) 알고리즘과 TCP_NODELAY
- 네이글 알고리즘은 네트워크의 효율을 위해 패킷을 전송하기 전에 많은 양의 TCP 데이터를 한 개의 덩어리로 합치는 알고리즘이다. (O/X)
- 네이글 알고리즘은 작은 메시지의 경우 패킷을 채우지 못해 기다리며 지연되는 단점이 있는데, 확인응답 지연 알고리즘과 함께 사용하면 이를 보완할 수 있다. (O/X)
- 네이글 알고리즘은 HTTP 스택에 TCP_NODELAY 파라미터 값을 설정하면 비활성화 시킬 수 있다. (O/X)

### 4.2 　 TCP의 성능에 대한 고려　 `yeosong`
#### 4.2.1 HTTP 트랜잭션 지연
- HTTP 지연은 클라이언트나 서버가 너무 많은 데이터를 내려받거나 복잡하고 동적인 자원들을 실행할 때 생긴다. (O)
- HTTP 지연의 대부분은 TCP 네트워크 지연 때문에 생긴다. (O)
- TCP 네트워크의 지연 정도는 요청/응답 메시지의 크기, 클라이언트와 서버간의 거리에 따라서도 달라진다. (O)

#### 4.2.2 성능 관련 중요 요소

##### 4.2.3 TCP 커넥션 핸드셰이크 지연
- 아주 큰 데이터를 주고받지 않는 평범한 HTTP 트랜잭션은 50% 이상의 시간을 TCP를 구성하는 데 쓴다. (O)
- TCP 구성으로 인한 지연을 제거하기 위해 이미 존재하는 커넥션을 재활용하기도 한다. (O)

##### 4.2.4 확인응답 지연
- TCP는 성공적인 데이터 전송을 보장하기 위해 확인응답 패킷을 사용한다. (O)
- '확인응답 지연' 알고리즘은 확인응답 패킷을 잠시 버퍼에 둔 채로 편승(piggyback)할 수 있는 데이터 패킷를 찾는 알고리즘이다. (O)

##### 4.2.5 TCP 느린 시작(slow start)
- 인터넷의 급작스러운 부하와 혼잡을 방지하기 위하여 커넥션의 최대 속도를 제한하여 점차 높여 나가는 것을 'silly window syndrome'이라고 부른다. (X)
  - TCP 느린 시작이라고 한다. 
  - silly window syndrome 전송자는 1바이트 크기의 패킷을 대량으로 전송하는 비효율적, 반사회적인 전송자를 말한다. 
- 패킷이 성공적으로 전달되는 각 시점에 송신자는 추가로 4개의 패킷을 더 전송할 수 있는 권한을 얻는다. (X)
  - 2개씩 늘어난다. 이를 'congestion window를 연다'고 말한다.
- 새로운 커넥션은 항상 튜닝된 커넥션보다 빠르다 (X)
  - TCP는 언제나 느린 시작을 하므로, 느린 시작 이후 튜닝된 커넥션이 언제나 새로운 커넥션보다 빠르다.

##### 4.2.6 네이글(Nagle) 알고리즘과 TCP_NODELAY
- 네이글 알고리즘은 네트워크의 효율을 위해 패킷을 전송하기 전에 많은 양의 TCP 데이터를 한 개의 덩어리로 합치는 알고리즘이다. (O)
- 네이글 알고리즘은 작은 메시지의 경우 패킷을 채우지 못해 기다리며 지연되는 단점이 있는데, 확인응답 지연 알고리즘과 함께 사용하면 이를 보완할 수 있다. (X)
  - 97p 함께 쓰일 경우 형편없이 동작한다. 네이글은 확인응답을 기다리고, 확인응답 지연은 확인응답을 지연시킨다. 
- 네이글 알고리즘은 HTTP 스택에 TCP_NODELAY 파라미터 값을 설정하면 비활성화 시킬 수 있다. (O)

    - 지속커넥션인지 확인하기 전까지는 파이프라인을 이어선 안된다.
2. http 클라이언트는 지속 커넥션 응답에 대해 요청 순서와 상관없이 Content-Length을 통해 정렬하여 해석할 수 있다. (O/X)
  - 정답 : X
    - http 응답은 요청 순서와 같게 와야한다. http 메세지는 순번이 매겨져 있지 않아서 응답이 순서없이 오면 순서에 맞게 정렬 시킬 방법이 없다.
3. http 지속 커넥션의 요청에 대한 서버의 응답 메세지가 어떠한 오류로 끊어지더라도 서버에서 이를 기억하여 다시 보내준다. (O/X) 
  - 정답 : X
    - http 클라이언트는 커넥션이 언제 끊어지더라도 완료되지 않은 요청이 파이프라인에 있으면 언제든 다시 요청을 보낼 준비가 되어야 한다.
4. 모든 메서드는  파이프라인 커넥션을 사용할 수 있다. (O/X)
  - 정답 : X
    - 멱등한 메서드만 사용할 수 있다. 비멱등(nonidempotent)한 메서드를 사용하여 에러가 발생하면 파이프라인을 통한 요청 중 어떤 것들이 서버에서 처리가 되었는지 알 수 없기 때문에 요청을 재차 보내면 문제가 생길 수 있다. 
5. (+a) 현재 HTTP/1.1 파이프라이닝은 모던 브라우저에서 기본적으로 활성화되어 사용되고 있다. (O/X)
- 정답 : O
  - HTTP 파이프라이닝은 모던 브라우저에서 기본적으로 활성화되어있지 않다.
  - 왜냐하면 정확히 구현해내기 어렵기 때문이다. 전송 중인 리소스의 크기, 효과적인 RTT(Round-trip delay, 왕복지연시간),  대역폭 크기의 설정, HOL(Head of Line Blocking, 헤드라인차단, 첫 번째 패킷이 응답이 올 때 까지, 이전 요청이 완료 될 때 까지 기다리는)으로 문제가 된다.
  - 파이프라이닝은 HTTP/2로 오면서 멀티플렉싱으로 대체 되어 사용된다.
  - 한 줄 요약 : **파이프라인 커넥션은 잘 사용하지 않는다.** ~(왜 공부한거니)~
- https://developer.mozilla.org/ko/docs/Web/HTTP/Connection_management_in_HTTP_1.x#HTTP_파이프라이닝

### 4.7 　 커넥션 끊기에 대한 미스터리　 `kukim`
1. HTTP 커넥션은 반드시 메세지를 다 보낸 후 종료된다. (O/X) 
- 정답 : X
  - 보통 커넥션은 메세지를 다 보낸 다음 끊지만 종종, 자주 에러가 있는 상황에서 헤더의 중간에서 끊거나 엉뚱한 곳에서 잘리기도 한다.
2. Content-Length 헤더의 역할은 무엇인가? (주관식)
  - 실제 받아야하는 엔터티와 실제로 전달받은 엔터티의 차이를 확인하는 헤더이다. 클라이언트나 프락시 서버가 커넥션이 끊어졌다는 HTTP 응답을 받은 후, 실제 전달된 엔터티의 길이와 Content-Length의 값이 일치하지 않거나 Content-Length 값 자체가 존재하지 않으면 수신다는 데이터의 정확한 길이를 서버에게 물어봐야 한다.
3. HTTP/1.1 명세에서 언제, 어떻게 커넥션을 끊는가에 대한 명확한 기준이 정해져있다. (O/X)
- 정답 : X
  - ‘우아하게 커넥션을 끊어야 한다’ 라고만 적혀있다. (issue a graceful close on the transport connection)  
4. `우아하게`
  - ‘우아하게 커넥션을 끊는 방법 중’ 클라이언트가 예기치 않은 상황에서 커넥션을 끊어야 할 때 절반 끊기 중 (입력/출력 채널)을 끊는다.  
    - (출력채널)을 끊어야 한다. 입력 채널을 끊는다면 이 전에 요청한 커넥션에 대한 결과를 서버에서 응답받을 수 없기 때문이다.
  - 절반 끊기 (입력/출력 채널)을 끊은 후 우아하게 기다린다. (O/▲/X)  
    - 우아하게 기다리면서 데이터나 스트림의 끝을 식별하기 위해 입력 채널에 대한 상태 검사를 주기적으로 해야한다.
